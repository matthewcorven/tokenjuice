## 1. What tokenjuice is, in one paragraph

tokenjuice is a TypeScript (ESM, Node ≥ 20) library + CLI + per-host
adapter set that sits in front of an agent harness's shell tool and
shrinks the textual output before it gets shipped back to the LLM. The
original command always runs; tokenjuice only rewrites the **observed
output** (or, for Cursor, the shell input so that our own `wrap` handles
execution). Nothing is summarized by an LLM — reducers are deterministic
rules with regex/head/tail/filter/counter behavior. Artifacts, stats,
and rule verification are first-class surfaces.

Key slogans from the codebase that are worth internalizing:

- "library first, not framework-locked" — core is plain objects, no
  framework assumptions
- "no silent command rewrite" — `--raw` / `--full` is the explicit
  bypass; cursor's pre-tool wrap is the one deliberate exception
- two truncation boundaries (reducer vs capture) — `--raw` only bypasses
  the reducer; `--max-capture-bytes` tunes `wrap` capture ceiling
- rule precedence: builtin < `~/.config/tokenjuice/rules` < `.tokenjuice/rules`,
  overridden by rule `id`

## 3. CLI surface (`src/cli/main.ts`)

Shape: a single argv parser (`parseArgs`) feeding a `switch` in
`main()`. Subcommands:

| Command | Purpose | Notes |
|---|---|---|
| `reduce [file]` | Compact stdin or a file | `--classifier`, `--raw/--full`, `--store`, `--max-inline-chars`, `--trace` |
| `reduce-json [file]` | Machine-facing adapter surface | Reads direct payload or `{ input, options }` envelope; always emits JSON |
| `wrap -- <cmd>` | Spawn a child and reduce captured stdout/stderr | `--tee`, `--store`, `--raw`, `--max-capture-bytes` (default 4 MB) |
| `install {codex,claude-code,cursor,pi,opencode}` | Write host hook/extension | `--local` points hooks at current repo build |
| `uninstall {codex,opencode}` | Remove hook/extension (only these two are reversible here) | Others are currently unimplemented by design |
| `ls`, `cat <id>` | List / dump stored artifacts | Artifacts under `~/.tokenjuice/artifacts` |
| `verify [--fixtures]` | Validate rules (and fixture-pinned reducer output) | Exit 1 on failure |
| `discover`, `doctor`, `stats` | Analyze stored artifact metadata | Feed reducer coverage improvement |
| `doctor hooks` / `doctor <host>` | Inspect installed hook health | Aggregated + per-host views |
| Internal hook entrypoints | `codex-post-tool-use`, `claude-code-post-tool-use`, `cursor-pre-tool-use` | Invoked by installed hook scripts |

Flag quirks to note:

- `--raw` and `--full` are aliases for the same bypass flag.
- `--store-dir` overrides the default artifact dir for `ls`/`cat`/`reduce`/`wrap`.
- `--timezone` on `stats` accepts `local`, `utc`, or any IANA zone.
- `--wrap-launcher` is an internal flag used by the cursor runtime to
  locate the tokenjuice binary/JS entrypoint that should execute the
  wrapped command.

Library consumers who want to call the CLI as a subprocess use
`runReduceJsonCli` from `src/core/cli-client.ts` — spawns `tokenjuice
reduce-json`, pipes JSON in, parses JSON out, with timeout and output
size guards.

## 4. Core engine (`src/core/`)

Flow for a single reduction (`reduceExecution` in `reduce.ts`):

1. Load rules (bundled built-ins + optional user/project layers,
   compiled + regex-cached once per `cwd` in a module-level cache).
2. Build raw text from `combinedText` or `stdout+stderr`.
3. Normalize the `ToolExecutionInput` (`execution-input.ts` →
   `command.ts`): split argv from `command`, trim leading `cd`, unwrap
   shell launchers (`bash -lc '...'`), strip env assignments, surface
   "effective" argv for classification.
4. Derive match candidates (original / shell-body / effective) and pick
   the best rule via `resolveRuleMatch`/`findBestRuleMatch`
   (`classify.ts`). Scoring: priority × 1000, then argv/tool/include
   specificity, then candidate source priority, then id tiebreaker.
   `generic/fallback` is only used if nothing else matches.
5. Run the rule (`applyRule`): pretty-print JSON opt-in, normalize
   lines, strip ANSI, run `matchOutput` short-circuits, apply
   skip/keep filters, dedupe, trim edges, run formatter specializations
   (`git/status`, `git/diff`, `search/rg`, `cloud/gh`), compute
   counters, then head/tail summarize. Failure mode uses
   `rule.failure.*` for more generous head/tail retention.
6. Build inline output. For very small raw output we pass through;
   otherwise the compact summary wins. `formatInline` conditionally
   prepends exit-code line + fact strings (search, help, test families
   on failure, or whenever "omitted" appears in the compact text).
7. Assemble `CompactResult` with optional trace block and optional
   stored artifact ref when `store: true`.

Relevant helpers worth remembering:

- `core/command-shell.ts` — shell tokenization, compound detection,
  chain splitting.
- `core/command-match.ts` — shell-launcher unwrap and effective argv.
- `core/command-identity.ts` — name, git subcommand, "file content
  inspection" command classification (`cat`, `sed`, `head`, `tail`,
  `nl`, `bat`, `jq`, `yq`).
- `core/inventory-safety.ts` — policy for repo-inventory commands
  (`find`, `fd`, `ls`, `rg --files`, `git ls-files`, pipelines through
  `sort`/`head`/`tail`/`uniq`/`wc -l`). Surfaces
  `InspectionCommandSkipReason` used by host adapters.
- `core/integrations/compact-bash-result.ts` — the shared adapter
  primitive. Takes visible text + optional trusted full text, applies
  inspection policy, runs reduction, applies rewrite policy, and
  returns `{ action: "keep" | "rewrite", … }`. Every post-tool host
  adapter calls this (codex, claude-code, opencode, openclaw, pi).
- `core/integrations/rewrite-policy.ts` — guards that keep tokenjuice
  from making output *worse* (no compaction, low savings, weak generic
  fallback, compound commands skipping generic fallback).
- `core/reduce-formatters.ts`, `core/reduce-inspection-summary.ts`,
  `core/reduce-utils.ts` — specialized rewrites.
- `core/artifacts.ts` — file-backed raw text + metadata JSON in
  `~/.tokenjuice/artifacts` with private (0600/0700) perms.
- `core/analysis.ts` — powers `discover`, `doctor`, `stats`.
- `core/cli-client.ts` — CLI-as-IPC helper for library consumers.
- `core/json-protocol.ts` — validates the reduce-json request envelope.
- `core/fixtures.ts` — loads+verifies `*.fixture.json` tests.
- `core/rules.ts` — layered rule loading, validation, module cache.
- `core/validate-rules.ts` — schema + regex compile check.

## 5. Host adapters (`src/hosts/`)

All host adapters share three responsibilities: **install**, **doctor**,
and a **runtime hook** entrypoint. Each runtime ultimately calls the
same primitive (`compactBashResult`) with its own inspection policy and
payload shape. Five adapters exist today; each differs mainly in how it
reaches the agent's tool pipeline.

### 5.1 Codex (`hosts/codex/`)
- Post-tool compaction. Hook file: `~/.codex/hooks.json`
  (`CODEX_HOME` overrides).
- Runtime entry CLI subcommand: `codex-post-tool-use` (reads stdin
  JSON, writes JSON).
- Extra surface: reads `~/.codex/config.toml` `[features] codex_hooks`
  flag read-only; exposes status via `inspectCodexHooksFeatureFlag` +
  `parseCodexFeatureFlag`. Never rewrites that config.
- Keeps a `tokenjuice-hook.last.json` + rolling
  `tokenjuice-hook.history.jsonl` log next to `hooks.json` (capped at
  200 entries with a simple file lock for concurrent writes).
- `uninstall codex` removes tokenjuice's PostToolUse entries and leaves
  the rest of the config alone.
- Watches for stale Homebrew Cellar paths via shared `hook-doctor`.

### 5.2 Claude Code (`hosts/claude-code/`)
- Post-tool compaction. Settings file: `~/.claude/settings.json`
  (respects `CLAUDE_CONFIG_DIR`, then legacy `CLAUDE_HOME`).
- Updates `hooks.PostToolUse` with a `Bash` matcher pointing at
  `tokenjuice claude-code-post-tool-use`; preserves unrelated settings
  keys.
- No feature-flag surface; command-lookup logic mirrors codex (prefer
  installed tokenjuice on PATH unless `--local`).

### 5.3 Cursor (`hosts/cursor/`)
- **Pre-tool** shell rewriting (unique). Hooks file:
  `~/.cursor/hooks.json` (`CURSOR_HOME` overrides).
- Installs a `preToolUse.Shell` hook whose command is
  `<launcher> cursor-pre-tool-use --wrap-launcher <launcher>`.
  When the hook runs, `runCursorPreToolUseHook` rewrites the shell
  command to `tokenjuice wrap -- <host-shell> -lc '<original>'`.
- Host shell resolution order: `tool_input.shell` → `TOKENJUICE_CURSOR_SHELL`
  → `SHELL` → `sh`; leaves command untouched if none resolve.
- Native Windows (`process.platform === "win32"`) is explicitly denied
  in both install and runtime paths; users are nudged to WSL.
- Launcher nuance: if launcher path ends in `.js`, tokenjuice runs it
  via `node <path>.js` to avoid `+x` issues.

### 5.4 pi (`hosts/pi/`)
- Extension (not a plain hook). Lives at
  `~/.pi/agent/extensions/tokenjuice.js` (`PI_CODING_AGENT_DIR`
  overrides `.pi/agent`).
- Install process: bundle `src/hosts/pi/extension/runtime.ts` via
  esbuild (if `--local`) or read pre-bundled `dist/hosts/pi/extension/runtime.js`,
  then wrap it with generated module shell that calls
  `createTokenjuicePiExtension({ extensionCommand: "tj" })`.
- Extension registers `/tj` (status|on|off|raw-next|bypass-next),
  listens for `tool_result` events for the `bash` tool, respects pi's
  auto-compaction setting, reads optional "full output" file when pi
  truncated visible output, and emits `tokenjuice`-annotated details
  back on rewrite.
- Session state persisted via `pi.appendEntry("tokenjuice-pi-config", …)`.
- Removes a legacy sibling runtime file
  (`tokenjuice-runtime.js`) on install for backwards compat.

### 5.5 opencode (`hosts/opencode/`)
- Plugin (similar model to pi). Lives at
  `<opencode config dir>/plugins/tokenjuice.js`. Config dir lookup:
  `OPENCODE_CONFIG_DIR` → `XDG_CONFIG_HOME/opencode` → `~/.config/opencode`.
- Uses the opencode `tool.execute.after` hook; mutates `output.output`
  in place and stashes `tokenjuice` details on `output.metadata`.
- Same esbuild-bundled runtime shape as pi.
- Also supports `uninstall opencode`.

### 5.6 openclaw (`hosts/openclaw/`)
- Bundled-from-the-openclaw-side plugin, not installable via
  `tokenjuice install`. Only exports
  `createTokenjuiceOpenClawEmbeddedExtension()` from `hosts/openclaw/index.ts`.
- Runs as an openclaw extension receiving `tool_result` events for the
  `exec`/`bash` tool; emits compaction notice + details.
- Enabled user-side with
  `openclaw config set plugins.entries.tokenjuice.enabled true` on
  OpenClaw ≥ `2026.4.22`.
- Only host where the library exposes a secondary entry point via
  `package.json` `exports["./openclaw"]`.

### 5.7 Shared helpers (`hosts/shared/`)
- `hook-command.ts` — shell quoting, shell-word parser with Windows
  `"..."` handling, extraction of node/tokenjuice paths from a hook
  command string (used by doctor reports to check file existence).
- `hook-doctor.ts` — aggregate `doctorInstalledHooks` across codex +
  claude-code + cursor + pi (opencode + openclaw currently *not*
  included in the aggregate report; `doctor hooks` only covers the
  four). Status merge: `broken > warn > ok > disabled`.
- `tool-result.ts` — shared helpers for building
  compaction-notice strings and `tokenjuice` detail blobs used by pi,
  openclaw, and opencode.

## 6. Rule system and built-ins

- Schema: `src/rules/tokenjuice-rule.schema.json` — validated at load
  time (`validate-rules.ts`). Only fields documented in `docs/rules.md`
  are accepted (`additionalProperties: false` everywhere).
- Layered: built-in → user → project; override by `id`. `sortRules`
  keeps `generic/fallback` last so other rules win by order too.
- Built-in inventory (96 JSON rules today) covers these families:
  `archive`, `build`, `cloud`, `database`, `devops` (docker/kubectl),
  `filesystem`, `generic` (fallback + help), `git`, `install` (package
  managers), `lint`, `media`, `network`, `observability`, `openclaw`
  (sessions-history), `package` (apt/brew/dnf/yum), `search`, `service`,
  `system`, `task` (make/just), `tests`, `transfer`.
- Fixtures (`src/rules/fixtures/**/*.fixture.json`) pin reducer output
  and are exercised by `tokenjuice verify --fixtures`.
- Special-case code paths keyed off `rule.id` in `core/reduce.ts`:
  `git/status`, `git/diff`, `cloud/gh`, `search/rg`. If a new family
  needs a bespoke rewriter, it generally gets a branch in
  `reduce-formatters.ts` and a keyed call in `reduce.ts` rather than a
  new rule field.

Build pipeline: `pnpm generate:builtin-rules` walks `src/rules/**`, writes
`src/core/builtin-rules.generated.ts` as a list of JSON imports with the
node 20 JSON import assertion syntax. `pnpm build` then runs `tsc` plus
two esbuild passes (pi runtime, opencode runtime) and copies `src/rules`
into `dist/rules` so the published npm package can still find raw JSON
alongside the bundled constant.

## 7. Tests

`vitest.config.ts` drives Node-mode Vitest. Layout mirrors `src/`.

- `test/core/*.test.ts` — reducers, rule loading, classification, wrap
  behavior, artifact IO, CLI-as-IPC client, JSON protocol, inventory
  safety, trace output.
- `test/hosts/*.test.ts` — one file per host, plus a
  `codex-feature-flag.test.ts` slice. Heavy emphasis on temp-dir
  isolation and restoring env (`CODEX_HOME`, `CLAUDE_CONFIG_DIR`,
  `CURSOR_HOME`, `PI_CODING_AGENT_DIR`, `OPENCODE_CONFIG_DIR`,
  `TOKENJUICE_CURSOR_SHELL`, `SHELL`, `HOME`, `PATH`, `platform`).
- `test/hosts/shared/` — shared-helper tests.
- `test/scripts/repair-pnpm-bin-shims.test.ts` — covers the
  `prepare`-hook shim repair script.

Regression gate (from `docs/integration-playbook.md`) for any new host
work:

```
pnpm typecheck
pnpm vitest run test/hosts/<host>.test.ts
pnpm vitest run test/hosts/codex.test.ts test/hosts/claude-code.test.ts test/hosts/pi.test.ts
# plus core command/classify/trace tests if normalization paths change
```

`pnpm verify` runs `lint`, `lint:circular` (madge), `typecheck`, and
`test`. `pnpm verify:package` additionally builds, runs
`publint --strict`, and stages the release artifacts.

## 9. Conceptual model cheat sheet

Two axes you'll encounter repeatedly while reasoning about a new host:

```
                pre-tool rewrite        post-tool rewrite
file-based   │  cursor                  codex, claude-code
hook/config  │  (preToolUse + wrap)     (PostToolUse hook)
extension    │  — (none yet)            pi, opencode, openclaw
API/SDK      │                          (tool.execute.after etc)
```

Every runtime eventually reduces to:

```
compactBashResult({
  source, command, cwd?, visibleText, trustedFullText?,
  exitCode?, maxInlineChars?, inspectionPolicy,
  …rewrite policy thresholds…,
}) -> { action: "keep" | "rewrite", result?, … }
```

The host adapter's job is:

1. Detect the host's "shell-like tool ran and produced text" event.
2. Extract the command string and captured output.
3. Choose an `InspectionCommandPolicy` (today: every post-tool host uses
   `"allow-safe-inventory"`).
4. Optionally load a host-provided full-output file when the visible
   text was truncated.
5. Call `compactBashResult`.
6. On `rewrite`, replace the outgoing text and stash a tokenjuice
   details payload; on `keep`, leave the tool result alone.

That is the whole contract a new integration has to honor.

## 10. Known quirks / things to respect when extending

- `--raw` ≠ capture bypass. `wrap` still caps at
  `--max-capture-bytes` (default 4 MiB) regardless of `--raw`. Any new
  host pipeline must not assume "raw = unbounded".
- Host adapters must **not** reach into reducer internals. If they
  need new behavior, it belongs in `core/` (see `compactBashResult` as
  the canonical seam).
- Rule dispatch specializations keyed off `rule.id` in
  `core/reduce.ts` are the current convention for rewriters that are
  too structural for JSON rules. Adding another one is acceptable,
  inventing a new DSL field is not (per `docs/spec.md`).
- Tests leak global state unless every env var a host reads is
  snapshotted + restored. The existing host suites are the pattern.
- `doctor hooks` aggregate currently **does not** include opencode or
  openclaw. A new host should be added there explicitly when its
  doctor surface is meaningful (openclaw is deliberately excluded
  because it is owned by the openclaw project).
- `install` copy hardcodes a five-way list (`codex, claude-code, cursor,
  pi, opencode`). Error strings include that list — updating them is
  part of the install-new-host checklist.
- Cursor is the only host that can mutate the command before execution;
  that path runs through `tokenjuice wrap`, so wrap behavior is
  load-bearing for cursor. Tests treat it that way.

## 11. Open questions to answer *before* designing Copilot integration

Leaving these intentionally unanswered for now; they are the research
agenda that motivates the next survey doc:

1. **GitHub Copilot CLI (`@github/copilot` / `gh copilot`)**: which
   surface are we actually targeting — the standalone agentic CLI, the
   `gh copilot` extension, or both? Does either expose a pre- or post-
   tool-use hook, an MCP server, an extension runtime, or only a raw
   transcript file? What is the stable scripting API today, and on what
   OSes?
2. **VS Code GitHub Copilot** (Copilot Chat + Agent Mode + tool calls):
   what extension points are usable today for intercepting tool output
   — `chat participant` API, `LanguageModelTool` return shaping, MCP
   servers registered via settings, or `tasks.json`/terminal profile
   shim? Any public hook for post-exec rewriting?
3. Config-dir and env-var conventions for both surfaces (to mirror the
   `CODEX_HOME` / `CURSOR_HOME` / `PI_CODING_AGENT_DIR` pattern).
4. Whether either surface supports "trusted full output on disk" (pi,
   opencode), or only shows truncated text to the model.
5. Windows story — VS Code runs natively on Windows, Copilot CLI likely
   does too; we'd need a plan that does not repeat cursor's "WSL only"
   boundary unless we consciously choose it.
6. Distribution: does a new integration need a separate npm package
   (e.g. a VS Code extension) or can it ride the existing
   `tokenjuice install <host>` flow?

Once those answers exist, the next step is a companion design doc in
this same `docs/_internal/` folder, then flipping bits of it into the
real docs (`README.md` table, `docs/spec.md` supported hosts, a
dedicated design doc per surface when behavior differs materially).
